# Creating Sensor Controller Project Files (.scp)

This guide provides complete, error-free rules for generating `.scp` (Sensor Controller project) XML files. Every element, attribute, and value is derived from analysis of validated examples.

## Overview

The `.scp` file is an XML project configuration for **Sensor Controller Studio (SCS)**. It defines:
- **Project metadata** (chip, board, operating system)
- **One or more tasks** with SCCode (Sensor Controller source code)
- **Resource allocations** (ADC, UART, GPIO, I²C, timers, etc.)
- **Data structures** for communication between System CPU and Sensor Controller (cfg, input, output, state)

**Critical Rule:** The `.scp` is the ONLY file you author. All other files (`scif.c`, `scif.h`, framework code) are generated by Sensor Controller Studio. Do NOT hand-edit any generated files.

---

## 1. XML Declaration and Project Root

### Structure

```xml
<?xml version="1.0"?>
<project name="Human-Readable Project Name" version="2.0.0.0">
    <!-- Project attributes and tasks go here -->
</project>
```

### Rules

| Element | Attribute | Value | Required | Notes |
|---------|-----------|-------|----------|-------|
| XML declaration | (none) | `<?xml version="1.0"?>` | ✓ | Always first line. Optional: add `encoding="UTF-8"` |
| `<project>` | `name` | Human-readable string | ✓ | Example: "ADC Data Logger for LaunchPad" |
| `<project>` | `version` | `2.0.0.0` | ✓ | Fixed version number; do not change |

### Examples

**Minimal:**
```xml
<?xml version="1.0"?>
<project name="My Sensor Task" version="2.0.0.0">
```

**With encoding:**
```xml
<?xml version="1.0" encoding="UTF-8"?>
<project name="My Sensor Task" version="2.0.0.0">
```

## 2. Project Description

### Structure

```xml
<project ...>
    <desc><![CDATA[One or more paragraphs describing the project.

    This can span multiple lines and include details about what the SC
    does, how it communicates with the System CPU, and any special features.]]></desc>
    <!-- Rest of project follows -->
</project>
```

### Rules

- Use `<desc>` element immediately after `<project>` opening tag
- Wrap text in `<![CDATA[ ... ]]>` to avoid XML escaping
- Multi-paragraph; separate paragraphs with blank lines
- Describe the functional purpose, not internal implementation
- Can reference example applications in `main.c` or `main_tirtos.c`

### Examples

**Single paragraph:**
```xml
<desc><![CDATA[Demonstrates periodic LED blinking driven by the Sensor Controller.]]></desc>
```

**Multi-paragraph with references:**
```xml
<desc><![CDATA[Samples an ADC channel periodically and stores values in a circular buffer.

The System CPU application reads the buffer head index to detect new samples and logs the data over UART.

See the header in the application source file ("main.c" or similar) for further details and instructions. This file is located in the source code output directory.]]></desc>
```

## 3. Project Attributes (pattr)

### Structure

```xml
<project ...>
    <desc> ... </desc>
    <pattr name="Apply default power mode">0</pattr>
    <pattr name="Board">CC26X2R1 LaunchPad</pattr>
    <pattr name="Chip name">CC2642R1F</pattr>
    <!-- ... more pattr entries ... -->
</project>
```

### Rules

**All 12 `pattr` entries are REQUIRED.** Do not omit any. The exact names and order below are mandatory.

| Order | Name | Valid Values | Example | Notes |
|-------|------|--------------|---------|-------|
| 1 | `Apply default power mode` | `0` | `0` | Literal value; always 0 |
| 2 | `Board` | Board name from device list | `CC26X2R1 LaunchPad` | Match to chip; see device list |
| 3 | `Chip name` | Chip part number | `CC2642R1F` | E.g., CC2642R1F, CC1312R1, CC1352R1 |
| 4 | `Chip package` | Package code | `QFN48 7x7 RGZ` | Match to chip |
| 5 | `Chip revision` | Revision letter | `E` | E.g., E, F, G (device-dependent) |
| 6 | `Clock in active mode` | `0` | `0` | Literal value; always 0 |
| 7 | `Code prefix` | Empty or prefix string | `` (empty) | Leave empty unless SCS generates with prefix |
| 8 | `Default power mode` | `0` | `0` | Literal value; always 0 |
| 9 | `Operating system` | `None`, `TI-RTOS`, `TI-DPL` | `TI-RTOS` | Default to TI-RTOS unless bare-metal required |
| 10 | `Output directory` | Path | `./source` | Relative path where SCS generates files |
| 11 | `RTC tick interval (16.16 format)` | Integer | `2000` | 16.16 fixed-point; 2000 = ~30.5 ms |
| 12 | `Run-time logging UART RX pin` | Pin name or empty | `` (empty) | Leave empty; populated only for runtime logging projects |
| 13 | `Run-time logging UART TX pin` | Pin name or empty | `` (empty) | Leave empty; populated only for runtime logging projects |

### Attribute Structure

Each `pattr` has the structure:
```xml
<pattr name="Exact Name From Table">value</pattr>
```

**Name must be EXACT** — spacing, capitalization, punctuation all matter.

### Chip and Board Compatibility

Match `Chip name`, `Chip package`, and `Chip revision` to the `Board` value:

| Board | Chip Name | Package | Revision |
|-------|-----------|---------|----------|
| CC26X2R1 LaunchPad | CC2642R1F | QFN48 7x7 RGZ | E |
| CC1312R1 LaunchPad | CC1312R1F | QFN48 7x7 RGZ | E |
| CC1352R1 LaunchPad | CC1352R1F | QFN48 7x7 RGZ | E |
| (Consult 00_overview.md for full device matrix) | ... | ... | ... |

### Examples

**Standard TI-RTOS project:**
```xml
<pattr name="Apply default power mode">0</pattr>
<pattr name="Board">CC26X2R1 LaunchPad</pattr>
<pattr name="Chip name">CC2642R1F</pattr>
<pattr name="Chip package">QFN48 7x7 RGZ</pattr>
<pattr name="Chip revision">E</pattr>
<pattr name="Clock in active mode">0</pattr>
<pattr name="Code prefix"></pattr>
<pattr name="Default power mode">0</pattr>
<pattr name="Operating system">TI-RTOS</pattr>
<pattr name="Output directory">./source</pattr>
<pattr name="RTC tick interval (16.16 format)">2000</pattr>
<pattr name="Run-time logging UART RX pin"></pattr>
<pattr name="Run-time logging UART TX pin"></pattr>
```

## 4. Tasks

### Structure

```xml
<project ...>
    <desc> ... </desc>
    <pattr> ... </pattr>
    <pattr> ... </pattr>
    <!-- All pattr entries -->
    
    <task name="Task Name">
        <!-- Task content: desc, tattr, resource_ref, sccode, etc. -->
    </task>
    
    <task name="Another Task">
        <!-- Second task content -->
    </task>
</project>
```

### Rules

- One or more `<task>` elements per project
- Each `<task>` has a `name` attribute (human-readable, unique within project)
- Task order in XML is preserved and becomes the order in the generated code

### Example

```xml
<task name="ADC Data Logger">
    <!-- Task internals -->
</task>

<task name="System CPU Alert Handler">
    <!-- Task internals -->
</task>
```

## 5. Task Description (task-level `<desc>`)

### Structure

```xml
<task name="Task Name">
    <desc><![CDATA[Brief description of what this task does.

    Additional details about behavior, interaction with the System CPU,
    or other important notes.]]></desc>
    <!-- Rest of task follows -->
</task>
```

### Rules

- Immediately after `<task>` opening tag
- Wrap in `<![CDATA[ ... ]]>`
- One or more paragraphs
- Describe the task's purpose and behavior, not internal code

### Example

```xml
<desc><![CDATA[Toggles the LED at a fixed rate using RTC-based scheduling.

The blink interval is BLINK_INTERVAL RTC ticks. With the default ~30.5 ms tick, a value of 16 produces approximately a 2 Hz blink rate (~488 ms on, ~488 ms off).]]></desc>
```

## 6. Task Attributes (tattr)

### Structure

```xml
<task name="...">
    <desc> ... </desc>
    <tattr name="CONSTANT_NAME" desc="..." type="dec" content="const" scope="task" min="0" max="65535">128</tattr>
    <tattr name="output.head" desc="..." type="dec" content="struct" scope="task" min="0" max="65535">0</tattr>
    <tattr name="output.pSamples" desc="..." size="CONSTANT_NAME" type="dec" content="struct_array" scope="task" min="0" max="65535">0</tattr>
    <tattr name="state.counter" desc="..." type="dec" content="struct" scope="task" min="0" max="65535">0</tattr>
    <!-- More tattr entries -->
</task>
```

### Rules

**Each `tattr` (task attribute) defines:**
- A compile-time constant (prefixed with all-caps, e.g., `BUFFER_SIZE`)
- A struct field (prefixed with struct name: `cfg.`, `input.`, `output.`, `state.`)
- An array field (with `size="CONSTANT"` reference)

| Attribute | Values | Required | Notes |
|-----------|--------|----------|-------|
| `name` | Identifier | ✓ | `CONSTANT_NAME`, `output.field`, `cfg.field`, etc. |
| `desc` | Human-readable | ✓ | Short description of purpose |
| `type` | `dec` | ✓ | Literal value; always `dec` (16-bit decimal integer) |
| `content` | `const`, `struct`, `struct_array` | ✓ | `const` = compile-time constant; `struct` = scalar struct field; `struct_array` = array |
| `scope` | `task` | ✓ | Literal value; always `task` |
| `min` | Integer | ✓ | Minimum valid value; typically `0` |
| `max` | Integer | ✓ | Maximum valid value; typically `65535` (16-bit unsigned) |
| `size` | Constant name | ✗ | Required only for `struct_array` content; references a `const` tattr name |
| Text content | Integer | ✓ | Initial/default value (integer from 0 to max) |

### Naming Conventions

**Constants (compile-time):**
- All uppercase with underscores: `BUFFER_SIZE`, `SAMPLE_COUNT`, `THRESHOLD_VALUE`
- Referenced by `size="CONSTANT"` in array definitions

**Struct fields:**
- Format: `struct_name.field_name`
- Struct names: `cfg`, `input`, `output`, `state`
- Field names: camelCase (e.g., `head`, `counter`, `pSamples`)

**Special naming patterns:**
- `output.head` — head index of circular buffer
- `output.pSamples` — pointer-like array: `pSamples[BUFFER_SIZE]`
- `state.counter` — internal state incremented each execution

### Examples

**Constant (compile-time):**
```xml
<tattr name="BUFFER_SIZE" desc="ADC sample buffer size" type="dec" content="const" scope="task" min="0" max="65535">128</tattr>
```

**Output struct scalar:**
```xml
<tattr name="output.head" desc="Circular buffer head index" type="dec" content="struct" scope="task" min="0" max="65535">0</tattr>
```

**Output struct array (size references BUFFER_SIZE constant):**
```xml
<tattr name="output.pSamples" desc="ADC sample buffer" size="BUFFER_SIZE" type="dec" content="struct_array" scope="task" min="0" max="65535">0</tattr>
```

**State struct field:**
```xml
<tattr name="state.counter" desc="Incremented once per task execution" type="dec" content="struct" scope="task" min="0" max="65535">0</tattr>
```

### Multiple Tasks with Shared Structs

When multiple tasks access the same struct (e.g., both read `output.counter`):
- Define the struct fields in the first task that uses them
- Do NOT re-declare the same struct field in a second task; reference it only in SCCode

Example (two tasks in one project):
```xml
<task name="LED Blinker">
    <desc> ... </desc>
    <tattr name="output.counter" desc="Incremented each alert" type="dec" content="struct" scope="task" min="0" max="65535">0</tattr>
    <tattr name="state.counter" desc="Incremented per execution" type="dec" content="struct" scope="task" min="0" max="65535">0</tattr>
    <!-- ... resource_ref and sccode ... -->
</task>

<task name="UART Emulator">
    <desc> ... </desc>
    <!-- No tattr re-declaration; just access output.counter in SCCode -->
    <sccode name="execute" init_power_mode="0"><![CDATA[
        // Access the counter from LED Blinker task
        uartWriteData(output.counter);
    ]]></sccode>
</task>
```

**Critical:** Struct fields are GLOBAL across all tasks in a project. Define them once; reference everywhere.

### Rules for All Resources

1. **Name must be EXACT** — match character-for-character (including spaces)
2. **All 32 resources are REQUIRED** in every task
3. **Order is preserved** from the list above — do not reorder
4. When `enabled="0"`, include nested elements only for resources with fixed sub-elements (see below)
5. When `enabled="1"`, populate `<io_usage>` and `<rattr>` elements as needed

### Resources with Nested Elements

These resources have **required sub-elements that must be present even when `enabled="0"`:**

| Resource | Sub-Elements |
|----------|--------------|
| I2C Master | `<rattr>` (SCL frequency, SCL stretch timeout), `<io_usage>` (pin references) |
| Multi-Buffered Output Data Exchange | `<rattr>` (buffer count, overflow settings) |
| SPI Data Transfer | `<rattr>` (bit rate, pin configs), `<io_usage>` (MISO, MOSI, SCLK pins) |
| UART Emulator | `<rattr>` (buffer sizes, idle period), `<io_usage>` (RX, TX pins) |
| Digital Output Pins | `<io_usage>` (pin array) |
| Analog Pins | `<io_usage>` (pin array) |
| LCD Controller | `<rattr>` (waveform state count), `<io_usage>` (pin array) |

### Pin Arrays

For resources with pin arrays (e.g., Digital Output Pins, Analog Pins), the structure is:

```xml
<io_usage name="USAGE_NAME" label="Human-readable description">
    <uattr name="Pin count">0</uattr>
    <uattr name="Pin/0000">DIO6</uattr>
    <uattr name="Pin/0001"></uattr>
    <uattr name="Pin/0002"></uattr>
    <!-- ... Pin/0003 through Pin/XXXX, all empty unless used ... -->
</io_usage>
```

**Rules for pin arrays:**
- `name` attribute: Uppercase identifier (e.g., `LED`, `SENSOR_OUTPUT`, `I2C_PINS`)
- `label` attribute: Human-readable description
- `Pin count` is always `0` (SCS manages this internally)
- First pin (`Pin/0000`) is the ONLY one populated with a DIO name if used
- All other pin slots are present but empty `<uattr name="Pin/XXXX"></uattr>`
- DIO names: `DIO0` through `DIO31` (exact format, uppercase)

### Examples

**ADC with Analog input:**
```xml
<resource_ref name="ADC" enabled="1"/>
<resource_ref name="Analog Pins" enabled="1">
    <io_usage name="SENSOR_OUTPUT" label="Analog sensor output">
        <uattr name="Pin count">0</uattr>
        <uattr name="Pin/0000">DIO23</uattr>
        <uattr name="Pin/0001"></uattr>
        <uattr name="Pin/0002"></uattr>
        <uattr name="Pin/0003"></uattr>
        <uattr name="Pin/0004"></uattr>
        <uattr name="Pin/0005"></uattr>
        <uattr name="Pin/0006"></uattr>
        <uattr name="Pin/0007"></uattr>
    </io_usage>
</resource_ref>
```

**Digital Output (LED pin):**
```xml
<resource_ref name="Digital Output Pins" enabled="1">
    <io_usage name="LED" label="Red LED on LaunchPad">
        <uattr name="Configuration on uninitialization">-1</uattr>
        <uattr name="Output drive strength">0</uattr>
        <uattr name="Output value on initialization">0</uattr>
        <uattr name="Pin count">0</uattr>
        <uattr name="Pin/0000">DIO6</uattr>
        <uattr name="Pin/0001"></uattr>
        <!-- ... Pin/0002 through Pin/0031 (32 total pin slots) ... -->
        <uattr name="Pin/0031"></uattr>
    </io_usage>
</resource_ref>
```

**UART Emulator with pins:**
```xml
<resource_ref name="UART Emulator" enabled="1">
    <rattr name="RX buffer size">64</rattr>
    <rattr name="Required idle period before enabling RX">11</rattr>
    <rattr name="TX buffer size">64</rattr>
    <io_usage>
        <uattr name="RX pin/0000">DIO2</uattr>
        <uattr name="TX pin/0000">DIO3</uattr>

```

**I2C Master (not used, but includes default structure):**
```xml
<resource_ref name="I2C Master" enabled="0">
    <rattr name="SCL frequency">0</rattr>
    <rattr name="SCL stretch timeout">1</rattr>
    <io_usage>
        <uattr name="SCL pin/0000"></uattr>
        <uattr name="SDA pin/0000"></uattr>
    </io_usage>
</resource_ref>
```

## 8. SCCode Blocks

### Structure

```xml
<task name="...">
    <!-- desc, tattr, resource_ref -->
    
    <sccode name="event0" init_power_mode="0"><![CDATA[]]></sccode>
    <sccode name="event1" init_power_mode="0"><![CDATA[]]></sccode>
    <sccode name="event2" init_power_mode="0"><![CDATA[]]></sccode>
    <sccode name="event3" init_power_mode="0"><![CDATA[]]></sccode>
    <sccode name="execute" init_power_mode="0"><![CDATA[
        // SCCode for execute block
        adcEnableSync(...);
        adcReadFifo(...);
    ]]></sccode>
    <sccode name="initialize" init_power_mode="0"><![CDATA[
        // SCCode for initialize block
        fwScheduleTask(1);
    ]]></sccode>
    <sccode name="terminate" init_power_mode="0"><![CDATA[
        // SCCode for terminate block (often empty)
    ]]></sccode>
</task>
```

### Rules

**All seven `sccode` blocks are REQUIRED** in every task, even if some are empty.

| Block | Purpose | When Run | Typical Content |
|-------|---------|----------|-----------------|
| `event0` | External trigger handler | When GPIO/comparator event 0 fires | Usually empty if event0 not used |
| `event1` | External trigger handler | When GPIO/comparator event 1 fires | Usually empty if event1 not used |
| `event2` | External trigger handler | When GPIO/comparator event 2 fires | Usually empty if event2 not used |
| `event3` | External trigger handler | When GPIO/comparator event 3 fires | Usually empty if event3 not used |
| `execute` | Main task code | On RTC trigger or manual schedule | Core functionality (ADC read, GPIO toggle, etc.) |
| `initialize` | Setup code | Once when task starts | Enable resources, set initial state, schedule first execution |
| `terminate` | Cleanup code | When task stops | Often empty; disable resources if needed |

### Attributes

Each `<sccode>` has:
- `name` attribute: One of `event0`, `event1`, `event2`, `event3`, `execute`, `initialize`, `terminate`
- `init_power_mode` attribute: Always `"0"` (literal value)
- Text content: SCCode wrapped in `<![CDATA[ ... ]]>`

### SCCode Content

**SCCode rules** (brief; see `01_sccode_language.md` for complete syntax):
- Variables: `U16` type (16-bit unsigned)
- Functions: `adcEnableSync()`, `adcGenManualTrigger()`, `gpioSetOutput()`, `fwScheduleTask()`, etc.
- Constants: `AUXIO_A_SENSOR_OUTPUT`, `ADC_REF_FIXED`, etc. (auto-generated from `io_usage` names)
- Comments: C-style `/* ... */` or `// ...`
- Statements end with semicolon
- No return statements (tasks don't return)

### Examples

**Empty event blocks (typical):**
```xml
<sccode name="event0" init_power_mode="0"><![CDATA[]]></sccode>
<sccode name="event1" init_power_mode="0"><![CDATA[]]></sccode>
<sccode name="event2" init_power_mode="0"><![CDATA[]]></sccode>
<sccode name="event3" init_power_mode="0"><![CDATA[]]></sccode>
```

**Execute block (ADC sampling):**
```xml
<sccode name="execute" init_power_mode="0"><![CDATA[// Enable ADC
adcEnableSync(ADC_REF_FIXED, ADC_SAMPLE_TIME_2P7_US, ADC_TRIGGER_MANUAL);

// Sample
U16 n = output.head;
adcGenManualTrigger();
adcReadFifo(output.pSamples[n]);
utilIncrAndWrap(n, BUFFER_SIZE, output.head);

// Disable ADC
adcDisable();

// Schedule next
fwScheduleTask(1);]]></sccode>
```

**Initialize block:**
```xml
<sccode name="initialize" init_power_mode="0"><![CDATA[// Select ADC input
adcSelectGpioInput(AUXIO_A_SENSOR_OUTPUT);

// Schedule first execution
fwScheduleTask(1);]]></sccode>
```

**Terminate block (cleanup):**
```xml
<sccode name="terminate" init_power_mode="0"><![CDATA[// Disable ADC
adcDisable();]]></sccode>
```

## 9. Task Metadata Elements

### Structure

```xml
<task name="...">
    <!-- desc, tattr, resource_ref, sccode blocks -->
    
    <event_trigger active_count="1">0,1,2,3</event_trigger>
    <tt_iter>run_execute</tt_iter>
    <tt_struct></tt_struct>
    <rtl_struct></rtl_struct>
    <rtl_task_sel en="1" struct_log_list="output"/>
</task>
```

### Rules

These four elements are **REQUIRED** for every task, but are generally fixed values derived from execution model.

| Element | Attributes / Content | Fixed Value | Notes |
|---------|----------------------|-------------|-------|
| `<event_trigger>` | `active_count` attribute | `"1"` | Always 1 for standard tasks |
| `<event_trigger>` | Text content | `0,1,2,3` | All four event triggers enabled |
| `<tt_iter>` | (none) | `run_execute` | Standard execution model |
| `<tt_struct>` | (none) | Empty or comma-separated fields | Usually empty; `output.field,state.field` if logging |
| `<rtl_struct>` | (none) | Empty | Usually empty |
| `<rtl_task_sel>` | `en` attribute | `"1"` | Always enabled |
| `<rtl_task_sel>` | `struct_log_list` attribute | `"output"` | Log the output struct |

### Examples

**Standard task (no internal struct logging):**
```xml
<event_trigger active_count="1">0,1,2,3</event_trigger>
<tt_iter>run_execute</tt_iter>
<tt_struct></tt_struct>
<rtl_struct></rtl_struct>
<rtl_task_sel en="1" struct_log_list="output"/>
```

**Task with multiple output fields to log:**
```xml
<tt_struct>output.counter,state.counter</tt_struct>
```

## 11. Validation Checklist

Use this checklist to verify your `.scp` before passing to Sensor Controller Studio:

### XML Structure
- [ ] First line is `<?xml version="1.0"?>`
- [ ] Root element is `<project name="..." version="2.0.0.0">`
- [ ] All tags are properly closed and nested

### Project Level
- [ ] `<desc>` immediately after `<project>` opening tag
- [ ] All 12 `<pattr>` entries present in correct order
- [ ] Chip/board compatibility verified

### Task Level (for each `<task>`)
- [ ] Task name is unique within project
- [ ] `<desc>` immediately after `<task>` opening tag
- [ ] All `<tattr>` entries use exact format: name, desc, type, content, scope, min, max, value
- [ ] Array `tattr` entries have `size="CONSTANT"` referencing a `const` tattr
- [ ] All 32 `<resource_ref>` entries present with exact names
- [ ] Resources with sub-elements have all required `<rattr>` and `<io_usage>` entries
- [ ] Pin arrays have full 8 or 32 pin slots (as applicable) with exact DIO names
- [ ] All 7 `<sccode>` blocks present (event0, event1, event2, event3, execute, initialize, terminate)
- [ ] SCCode wrapped in `<![CDATA[ ... ]]>`
- [ ] Four metadata elements present: `event_trigger`, `tt_iter`, `tt_struct`, `rtl_struct`, `rtl_task_sel`

### Content Verification
- [ ] No literal `<` or `>` characters in text (must use `<![CDATA[...]]>`)
- [ ] No attribute values missing quotes
- [ ] DIO names are uppercase: `DIO0` through `DIO31`, not `dio0` or `Dio0`
- [ ] Chip revision is single uppercase letter (E, F, G, etc.)
- [ ] Constant names in `tattr` and `size=` references are consistent

## 12. Error Prevention: Common Mistakes

| Mistake | Impact | Solution |
|---------|--------|----------|
| Omitted `<pattr>` entry | SCS parse error | Include all 12 pattr entries in exact order |
| Omitted `<resource_ref>` | SCS parse error | Include all 32 resources, even if `enabled="0"` |
| Wrong resource name (typo, spacing) | SCS parse error | Copy exact name from list; match case and spacing |
| Missing pin slots in array (e.g., only 1 pin defined) | SCS parse error | Include all 8 or 32 pin `<uattr>` slots |
| Unescaped `<` or `>` in SCCode | XML parse error | Wrap SCCode in `<![CDATA[...]]>` |
| Missing `<![CDATA[...]]>` around text | XML parse error | All text content must be in CDATA sections |
| Wrong `type` value (e.g., `"hex"` instead of `"dec"`) | SCS parse error | Always use `type="dec"` |
| Wrong `content` value (e.g., `"constant"` instead of `"const"`) | SCS parse error | Use exactly: `const`, `struct`, or `struct_array` |
| Lowercase DIO names (e.g., `dio23`) | SCS parse error / pin mismatch | Use uppercase: `DIO23` |
| Chip name mismatch (e.g., CC2642 instead of CC2642R1F) | SCS parse error / incompatible chip | Use exact part number from device matrix |
| Omitted SCCode block (e.g., missing `terminate`) | SCS parse error | All 7 sccode blocks required |
| Scope value other than `"task"` | SCS parse error | Always use `scope="task"` |
| Missing `init_power_mode` on `sccode` | SCS parse error | Always include `init_power_mode="0"` |

## 13. After Creating the .scp

Once your `.scp` file is error-free and validated:

1. **Open in Sensor Controller Studio** (GUI) or run **`scstudiocli --generate <project>.scp`** (CLI)
2. **SCS generates**:
   - `scif.c` — Bytecode and data structures
   - `scif.h` — SCIF driver API declarations
   - `scif_framework.c` / `scif_framework.h` — Framework (copy from SCS resources)
   - `scif_osal_*.c` / `scif_osal_*.h` — RTOS integration (TI-RTOS or TI-DPL variant)

3. **You then write**:
   - `main.c` or `main_tirtos.c` — System CPU application
   - `ex_include.h` — Header with configuration defines
   - CCS project files (`.projectspec`, etc.)

4. **Never hand-edit** any SCS-generated files.
