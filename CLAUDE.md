# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Purpose

This repository is a **reference documentation project** for the TI Sensor Controller (SC)—a dedicated ultra-low-power co-processor embedded in CC13xx/CC26xx wireless MCUs. The project provides comprehensive markdown documentation about the Sensor Controller architecture, programming model, resource APIs, and project setup.

The documentation is **not executable code**; it is reference material explaining:
- How the SC executes firmware while the main CPU sleeps
- SCCode language syntax and semantics
- SCIF (Sensor Controller Interface) driver API for System CPU communication
- Complete API catalog for SC resources (ADC, I²C, SPI, GPIO, Timers, UART, Comparators)
- 30+ example project patterns and their chip support matrix

The `examples/` directory contains reference SCCode task files (`.scp`), framework code, and bare-metal/RTOS example applications.

## Key Architecture Concepts

### Sensor Controller Context

The Sensor Controller is a **16-bit RISC co-processor** in the AUX (Always-On-AUX) power domain, separate from the System CPU (Arm Cortex-M). It has its own:
- **ISA**: Proprietary 16-bit RISC (not ARM)
- **Clock**: 24 MHz AUX oscillator
- **Memory**: 2–4 KB of AUX RAM (device-dependent)
- **Peripherals**: ADC, comparators, I²C, SPI, GPIO, timers (all in AUX domain)

The **System CPU can deep-sleep** while the SC runs, reducing average power consumption dramatically (0.5–4 µA typical for sensing tasks).

### Programming Model

1. **Sensor Controller firmware** is written in **SCCode** (a C-like DSL)
2. **Sensor Controller Studio** (required desktop tool) compiles SCCode → binary AUX RAM bytecode
3. **SCIF driver** generated by SCS provides the binary image (`pAuxRamImage[]`) and data struct definitions
4. **System CPU application** uses the SCIF API to start/stop SC tasks and communicate via shared AUX RAM

### Communication

- SC and System CPU communicate via **shared AUX RAM structures**: `cfg` (System CPU → SC), `input` (System CPU → SC), `output` (SC → System CPU), and `state` (SC internal)
- SC notifies System CPU of completion via **interrupt** (`fwGenAlertInterrupt()` in SCCode)
- System CPU reads results from the `output` struct in the ISR and resumes the SC with `scifAckAlertEvents()`

## Working with This Repository

### For Reference/Documentation Tasks

- All `.md` files are self-contained and can be read independently
- Cross-references use markdown `[link](file.md)` format
- Code examples in the `.md` files are pseudocode or simplified—refer to `examples/` for complete implementations

### For Example Code

Example files are located in `examples/`:
- **`.scp` files**: Sensor Controller task definitions (opened in Sensor Controller Studio, not edited by hand)
- **`source/`**: Framework files (`scif_framework.c/h`, `scif_osal_*.c/h`) and example applications
- **`projects/`**: CCS and IAR project files for importing into IDEs

> **Important**: The `.scp` file is an XML project file. It is never hand-edited. To modify SC firmware, open it in Sensor Controller Studio, edit the task blocks in the GUI, click "Generate Code", and copy the output `scif.c`/`scif.h` to your CCS project.

## Editing Guidelines

### Documentation Files

- Use markdown with clear code blocks for SCCode examples
- Use C++ style comments (`//`) inside `@code` blocks (not C-style `/* */`)
- Cross-reference with relative links: `[Architecture](00_overview.md)`
- Include concrete chip names (CC2640R2, CC1312R) rather than generic references
- Provide code snippets that show typical System CPU initialization and SC task structure

### Example Code

- Example C source files follow **TI coding standards** (see CLAUDE.md rules for connectivity projects)
- Framework files (`scif_framework.c/h`) are copied verbatim from the SCS installation—**do not edit**
- Generated files (`scif.c/.h` from SCS) are binary bytecode representations—**do not edit by hand**
- Application examples (like `main.c`, `main_tirtos.c`) show typical System CPU patterns and are editable

## External Tools Required

- **Sensor Controller Studio**: Desktop GUI for writing SCCode and generating `scif.c/h`. Download from [ti.com/tool/SENSOR-CONTROLLER-STUDIO](https://www.ti.com/tool/SENSOR-CONTROLLER-STUDIO)
- **Code Composer Studio (CCS)**: IDE for building and flashing System CPU firmware. Download from [ti.com/tool/CCSTUDIO](https://www.ti.com/tool/CCSTUDIO)
- **SimpleLink SDK**: One of CC2640R2, CC26x2, CC13x0, or CC13x2 SDK (device-specific)

There is **no Makefile or build system** in this repository—it is documentation. Projects are built in CCS.

## Workflow Reference

If a user wants to create a new project using this documentation:

1. **Read** `00_overview.md` to understand SC concepts
2. **Create SCCode task** in Sensor Controller Studio (see `05_project_setup.md`)
3. **Generate code** in SCS (produces `scif.c` and `scif.h`)
4. **Copy framework files** from SCS to CCS project (`scif_framework.c/h`, `scif_osal_*.c/h`)
5. **Write System CPU application** using SCIF API (see `03_scif_driver_api.md`)
6. **Build in CCS** and flash

See `04_examples_catalog.md` for reference implementations and `05_project_setup.md` for detailed setup instructions.

## Creating a New Sensor Controller Project (Authoring Rule)

**When the user asks Claude to "create a new Sensor Controller project" (or equivalent), Claude MUST produce only a single `.scp` XML file. Do NOT generate `scif.c`, `scif.h`, `scif_framework.c/.h`, `scif_osal_*.c/.h`, `main.c`, project specs, or any other artifact.**

Rationale: every file other than the `.scp` is produced by **Sensor Controller Studio** (GUI or CLI: `scstudiocli --generate <project>.scp`) by compiling the SCCode in the `.scp` to AUX RAM bytecode. The bytecode in `pAuxRamImage[]`, the AUX RAM addresses in `scifTaskData`, and the OSAL pairing in `scif.h` cannot be authored by hand — any hand-written copy will diverge from what SCS produces and will not run on hardware.

### `.scp` File Format

The `.scp` is XML. Use the canonical structure shown in the shipped `examples/adc_data_logger_launchpad.scp`:

```xml
<?xml version="1.0"?>
<project name="<Human-Readable Project Name>" version="2.0.0.0">
    <desc><![CDATA[<one or more paragraphs describing the project>]]></desc>

    <!-- Project-level attributes (pattr) -->
    <pattr name="Apply default power mode">0</pattr>
    <pattr name="Board">CC26X2R1 LaunchPad</pattr>
    <pattr name="Chip name">CC2642R1F</pattr>
    <pattr name="Chip package">QFN48 7x7 RGZ</pattr>
    <pattr name="Chip revision">E</pattr>
    <pattr name="Clock in active mode">0</pattr>
    <pattr name="Code prefix"></pattr>
    <pattr name="Default power mode">0</pattr>
    <pattr name="Operating system">TI-RTOS</pattr>           <!-- None | TI-RTOS | TI-DPL -->
    <pattr name="Output directory">./source</pattr>
    <pattr name="RTC tick interval (16.16 format)">2000</pattr>
    <pattr name="Run-time logging UART RX pin"></pattr>
    <pattr name="Run-time logging UART TX pin"></pattr>

    <task name="<Task Name>">
        <desc><![CDATA[<task description>]]></desc>

        <!-- Task attributes (tattr): constants, output/input/cfg/state struct fields -->
        <tattr name="BUFFER_SIZE" desc="..." type="dec" content="const" scope="task" min="0" max="65535">128</tattr>
        <tattr name="output.head"     desc="..." type="dec" content="struct"       scope="task" min="0" max="65535">0</tattr>
        <tattr name="output.pSamples" desc="..." size="BUFFER_SIZE" type="dec" content="struct_array" scope="task" min="0" max="65535">0</tattr>

        <!-- Resource references: enable resources used by this task; leave others enabled="0" -->
        <resource_ref name="ADC" enabled="1"/>
        <resource_ref name="Analog Pins" enabled="1">
            <io_usage name="SENSOR_OUTPUT" label="Analog sensor output">
                <uattr name="Pin count">0</uattr>
                <uattr name="Pin/0000">DIO23</uattr>
                <uattr name="Pin/0001"></uattr>
                <!-- ... up to Pin/0007 ... -->
            </io_usage>
        </resource_ref>
        <resource_ref name="Math and Logic" enabled="1"/>
        <resource_ref name="RTC-Based Execution Scheduling" enabled="1"/>
        <resource_ref name="System CPU Alert" enabled="0"/>
        <!-- ... include the full set of resource_ref entries (enabled="0" for unused) ... -->

        <!-- SCCode blocks. event0..event3, execute, initialize, terminate are mandatory -->
        <sccode name="event0"     init_power_mode="0"><![CDATA[]]></sccode>
        <sccode name="event1"     init_power_mode="0"><![CDATA[]]></sccode>
        <sccode name="event2"     init_power_mode="0"><![CDATA[]]></sccode>
        <sccode name="event3"     init_power_mode="0"><![CDATA[]]></sccode>
        <sccode name="execute"    init_power_mode="0"><![CDATA[<SCCode for Execute block>]]></sccode>
        <sccode name="initialize" init_power_mode="0"><![CDATA[<SCCode for Initialize block>]]></sccode>
        <sccode name="terminate"  init_power_mode="0"><![CDATA[<SCCode for Terminate block>]]></sccode>

        <event_trigger active_count="1">0,1,2,3</event_trigger>
        <tt_iter>run_execute</tt_iter>
        <tt_struct></tt_struct>
        <rtl_struct></rtl_struct>
        <rtl_task_sel en="1" struct_log_list="output"/>
    </task>
</project>
```

### Mandatory rules when authoring a `.scp`

1. **Header**: `<?xml version="1.0"?>` and `version="2.0.0.0"` on `<project>`.
2. **All `pattr` entries above are required**. Pick `Chip name` / `Chip package` / `Board` from the supported set in `00_overview.md`. Use TI-RTOS as the default `Operating system` unless the user asks for None or TI-DPL.
3. **`tattr` `content=` values**: `const` for compile-time constants, `struct` for `cfg/input/output/state` scalar fields, `struct_array` for arrays (size must reference a `const` `tattr` by name). `scope="task"` is the norm.
4. **`resource_ref`**: emit the full list of resources from the canonical example, marking the ones this task uses with `enabled="1"` and the rest with `enabled="0"`. Resources used in the SCCode (e.g. `adcEnableSync`, `i2cTx`, `gpioSetOutput`) must have their backing `resource_ref` enabled.
5. **`io_usage`**: each enabled pin-bearing resource needs an `io_usage` block naming the pin and assigning it to a DIO. The `name` becomes `AUXIO_*_<NAME>` in generated code.
6. **`sccode` blocks**: emit all seven (`event0..3`, `execute`, `initialize`, `terminate`) even if empty. SCCode goes inside `<![CDATA[...]]>`. Follow the SCCode rules in `01_sccode_language.md` (`U16` types, `fwScheduleTask`, `adcEnableSync`, etc.).
7. **Do not invent attributes** — if a field isn't in the canonical example, leave it out. SCS rejects unknown XML.

### After Claude writes the `.scp`

End the response by telling the user how to produce the rest of the project:

> Open this `.scp` in Sensor Controller Studio (or run `scstudiocli --generate <name>.scp`) to generate `scif.c`, `scif.h`, and the OSAL pair. Then copy `scif_framework.c/.h` from the SCS `resources/fw_templates/` folder and write your System CPU `main.c` against the SCIF API (see `03_scif_driver_api.md`).

Do not offer to write `main.c` or any of the generated files unless the user explicitly asks for them as a separate step.
