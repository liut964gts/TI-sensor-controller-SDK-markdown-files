# Project Setup Guide

How to create, build, and import a Sensor Controller project from scratch.

---

## Prerequisites

### 1. Install Sensor Controller Studio

Download from [ti.com/tool/SENSOR-CONTROLLER-STUDIO](https://www.ti.com/tool/SENSOR-CONTROLLER-STUDIO). Default install path:
```
C:\Program Files (x86)\Texas Instruments\Sensor Controller Studio\
```

### 2. Install the SimpleLink SDK

One of these SDKs is required. Download from [ti.com/wireless-connectivity](https://www.ti.com/wireless-connectivity):

| SDK | CCS Variable | Use for |
|---|---|---|
| SimpleLink CC2640R2 SDK | `COM_TI_SIMPLELINK_CC2640R2_SDK_INSTALL_DIR` | CC2640, CC2650, CC2640R2 |
| SimpleLink CC26x2 SDK | `COM_TI_SIMPLELINK_CC26X2_SDK_INSTALL_DIR` | CC2642R, CC2652R |
| SimpleLink CC13x0 SDK | `COM_TI_SIMPLELINK_CC13X0_SDK_INSTALL_DIR` | CC1310, CC1350 |
| SimpleLink CC13x2 SDK | `COM_TI_SIMPLELINK_CC13X2_SDK_INSTALL_DIR` | CC1312R, CC1352R |

### 3. Set the CCS SDK Variable

In CCS: **Window → Preferences → Code Composer Studio → Build → Variables**

Add the appropriate variable pointing to your SDK install directory, e.g.:
```
COM_TI_SIMPLELINK_CC2640R2_SDK_INSTALL_DIR = C:\ti\simplelink_cc2640r2_sdk_5_30_00_03
```

### 4. Install Code Composer Studio (CCS)

Version 10+ recommended. [ti.com/tool/CCSTUDIO](https://www.ti.com/tool/CCSTUDIO)

## Creating an SCP Project

1. Open Sensor Controller Studio.
2. **File → New** to create a new project.
3. In the project panel on the left:
   - Select **Device**: choose your chip (e.g., CC2650F128)
   - Select **Package**: e.g., QFN48 7×7
   - Set **Operating System**: None (OSAL None) for bare-metal
4. Click the **"+"** button on the Tasks panel to add a task.
5. In the task editor, fill in four code blocks: **Initialize**, **Execute**, **Terminate**, and optionally **Event Handlers**.
6. Select resources in the **Resources** panel (ADC, I²C, etc.). Each resource adds API functions to your task code.
7. Configure I/O pins in the **I/O Mapping** panel (assign DIO numbers to symbolic names like `AUXIO_A_SENSOR_INPUT`).

### SCP File Format

The `.scp` file is an XML file. You can open it in a text editor to see the full task code, resource configuration, and I/O mapping. It is not normally hand-edited.

## Generating the SCIF Driver

After writing SCCode:

1. In SCS, click **"Generate Code"** (toolbar button or **Project → Generate Code**).
2. SCS compiles the SCCode to AUX RAM bytecode and generates:
   - `scif.c` — contains `pAuxRamImage[]`, task data LUTs, resource helper functions
   - `scif.h` — contains task data structs, `scifTaskData` macro, `SCIF_TASK_DATA_ADDR`
3. Copy these two files to your CCS project's source directory.

> **Never hand-edit `pAuxRamImage[]`** — it is binary bytecode. Any manual change will produce undefined SC behavior.

### What Changes Between Generations

| Item | Location | Notes |
|---|---|---|
| `pAuxRamImage[]` | `scif.c` | Changes every time SCCode changes |
| `SCIF_TASK_DATA_ADDR` | `scif.h` | Changes when task struct layout changes |
| `pScifTaskDataStructInfoLut` | `scif.c` | Changes when struct sizes change |
| AUX RAM addresses in `scifDriverSetup` | `scif.c` | Changes when image layout changes |
| Task data struct typedef | `scif.h` | Changes when output/input/cfg/state fields change |

## File Organization

Recommended project layout:
```
my_project/
├── my_project.scp              ← Sensor Controller Studio project
├── source/
│   ├── main.c                  ← System CPU application
│   ├── scif.c                  ← Generated by SCS (copy here)
│   ├── scif.h                  ← Generated by SCS (copy here)
│   ├── scif_framework.c        ← Copied from SCS fw_templates/; edit one line
│   ├── scif_framework.h        ← Copied from SCS fw_templates/; verbatim
│   ├── scif_osal_none.c        ← Copied from SCS osal_defs/; verbatim
│   └── scif_osal_none.h        ← Copied from SCS osal_defs/; verbatim
└── ccs/
    └── my_project.projectspec  ← CCS import descriptor
```

### Copying Framework Files

From the SCS installation, copy these files to `source/`:

```
FROM: C:\Program Files (x86)\Texas Instruments\Sensor Controller Studio\
         resources\fw_templates\scif_framework__0.c   → source\scif_framework.c
         resources\fw_templates\scif_framework__0.h   → source\scif_framework.h
         resources\osal_defs\scif_osal_none__0.c      → source\scif_osal_none.c
         resources\osal_defs\scif_osal_none__0.h      → source\scif_osal_none.h
```

**Required edit in `scif_framework.c`**: Find this line (typically around line 41):
```c
#include "<<<osal_c_file>>>"
```
Replace with:
```c
#include "scif_osal_none.c"
```

This is the only edit needed. The rest of the file is used verbatim.

## CCS Project Setup

### Option A: Import from .projectspec

If you have a `.projectspec` file (like `ccs/adc_poll.projectspec` in this repo):

1. In CCS: **File → Import → Code Composer Studio → CCS Projects**
2. Browse to the folder containing the `.projectspec`
3. Tick the project and click **Finish**

### Option B: Create Manually

1. **File → New → CCS Project**
2. Select target: **CC2650F128** (or your device)
3. Connection: **XDS110**
4. Toolchain: **TI ARM Compiler**
5. Set compiler options:
   ```
   --silicon_version=7M3
   --little_endian
   --code_state=16
   --abi=eabi
   -me
   --gcc
   --c99
   --gen_func_subsections=on
   -DDeviceFamily_CC26X0
   -I${PROJECT_ROOT}
   -I${COM_TI_SIMPLELINK_CC2640R2_SDK_INSTALL_DIR}/source
   -I${COM_TI_SIMPLELINK_CC2640R2_SDK_INSTALL_DIR}/source/ti/devices/cc26x0
   ```
6. Set linker options:
   ```
   ${COM_TI_SIMPLELINK_CC2640R2_SDK_INSTALL_DIR}/source/ti/devices/cc26x0/driverlib/bin/ccs/driverlib.lib
   ```
7. Add SDK files to project (link, not copy):
   - `${SDK}/source/ti/devices/cc26x0/linker_files/cc26x0f128.cmd`
   - `${SDK}/source/ti/devices/cc26x0/startup_files/ccfg.c`
   - `${SDK}/source/ti/devices/cc26x0/startup_files/startup_ccs.c`
8. Add all files from `source/` to the project.
9. **Exclude `scif_osal_none.c` from build** (right-click → Properties → Build → Exclude from build). It is compiled inside `scif_framework.c`.

## Build and Flash

1. **Build** (Ctrl+B or **Project → Build All**)
2. Connect LaunchPad via USB
3. **Run → Debug** (F11) — this flashes the firmware and halts at `main()`
4. **Run → Resume** (F8) to execute

### Common Build Errors

| Error | Cause | Fix |
|---|---|---|
| `undefined reference to pAuxRamImage` | `scif.c` not in project | Add `scif.c` to project sources |
| `undefined symbol scifOsalRegisterCtrlReadyCallback` | Framework not in project | Add `scif_framework.c` |
| `duplicate symbol scifOsalEnterCriticalSection` | `scif_osal_none.c` compiled twice | Exclude `scif_osal_none.c` from direct build |
| `cannot open source file "ti/devices/cc26x0/..."` | SDK path wrong | Check `COM_TI_SIMPLELINK_CC2640R2_SDK_INSTALL_DIR` variable |
| `undefined reference to scifStartRtcTicksNow` | Wrong framework file | Ensure you're using project-specific `scif.c` (contains RTC functions) |

## Updating the SCIF Driver After SCCode Changes

Whenever you change SCCode in SCS:

1. Click **Generate Code** in SCS.
2. Overwrite `source/scif.c` and `source/scif.h` in your project with the new generated versions.
3. Rebuild in CCS.

The framework files (`scif_framework.c/h`, `scif_osal_none.c/h`) never need to change.

## Debugging with the SCS Simulator

Sensor Controller Studio includes a built-in simulator:

1. In SCS, open your project.
2. Click **Debug → Start Simulation**.
3. Step through SCCode blocks.
4. Inspect AUX RAM values in the **Registers** and **Variables** panels.
5. Set breakpoints on SCCode lines.

The simulator does not require hardware — useful for verifying logic before flashing.

## Power Optimization Tips

- Use `scifOsalDisableAuxDomainAccess()` between AUX RAM accesses to allow AUX to power down.
- Use `PRCMSleep()` in the System CPU main loop instead of a busy-wait spin.
- Reduce `ADC_SAMPLE_HZ` to lower duty cycle. At 1 Hz, the SC runs for ~50 µs every second.
- Use event-driven execution (EventHandler) instead of RTC scheduling when possible — SC only runs on actual events.
- Use `ADC_SAMPLE_TIME_2P7_US` (shortest) unless your sensor needs settling time.
- If you only need threshold detection (not the raw value), use the ADC Window Monitor resource — it generates zero alerts for in-range values.
